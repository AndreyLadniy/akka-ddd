Physical model of the Aggregate Root

The Aggregate Root instance is a Persistent Actor capable of handling Command Messages and writing Event Messages to the underlying journal. All Commands must contain an identifier of the business entity so that they can be routed to the appropriate Aggregate Root instance.
——–

The Akka-DDD framework allows to model the behavior of the AR as a state machine using the algebraic data type. Each type must extend from the AggregateActions trait and define a command processing logic consisting of two parts: command handling (reaction) and event handling (state transition). To do that, you need to construct an Actions object by calling handleCommand and then handleEvent methods as shown in the example below:


    def actions =
      handleCommand {
        case CreateLottery(id) => LotteryCreated(id)
      }
      .handleEvent {
        case LotteryCreated(_) => EmptyLottery
      }

Command handling (reaction)

The actual command handling logic (the Command Handler) is a partial function Command => Reaction that needs to be passed to the handleCommand method. The Reaction is either an Event (or a sequence of Events) or a Rejection. A couple of Command Handler examples below:

Command ⟶ Event

    case AddParticipant(id, name) => ParticipantAdded(name, id)

Command ⟶ sequence of Events (created manually)

    case Reset(id, name) =>
      NameChanged(id, name) & ValueChanged(id, 0, version + 1)

// Notice that `version` is a property of the current state.

Command ⟶ sequence of Events (being a result of an arbitrary sequence transformation)

    case RemoveAllParticipants(id) =>
      this.participants  
        .map { name => ParticipantRemoved(name, id) }

// Notice that `participants` is a property of the current state.

Command ⟶ Rejection (with a rejection reason of the default type: DomainException)

    case Run(_)  =>
      reject("Lottery has no participants")

Command ⟶ Rejection (with a rejection reason of a custom type)

    case anyCommand  =>
      reject (LotteryHasAlreadyAWinner(s"Lottery has already a winner and the winner is $winner"))

Command ⟶ Rejection or Event (conditional Rejection)

    case CreateDummy(id, name, description, value) =>
      rejectIf(value < 0, "negative value not allowed") orElse
        DummyCreated(id, name, description, value)
Collaboration with external actors

Sometimes the command handling logic is more complex and requires the AR Actor to collaborate with other Actors. In such cases the Reaction needs to be implemented as a method of the AR Actor and exposed as a property of type Reaction of the Aggregate Root configuration object. The command handler will simply return the Reaction that is available in the AR configuration.

Event handling (state transition)

    case ParticipantRemoved(name, id) =>
      val newParticipants = participants.filter(_ != name)
      // NOTE: if last participant is removed, transition back to EmptyLottery
      if (newParticipants.isEmpty)
        EmptyLottery
      else
        copy(participants = newParticipants)
